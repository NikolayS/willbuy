## ZFS + DBLab (PGDATA on ZFS, thin-clone branching) — issue #49

Spec ref: §A5 in `.samo/spec/willbuy/SPEC.willbuy.amendments.md`.

### Architecture overview

```
/dev/sda (80 GB OS disk)
└── /var/lib/zfs-tank/tank.img  (sparse loopback, up to 30 GB)
    └── ZFS pool: tank
        ├── tank/willbuy/pgdata  → /var/lib/postgresql/data   (recordsize=8K)
        └── tank/willbuy/pgwal   → /var/lib/postgresql/wal    (recordsize=128K)

Supabase docker-compose db service mounts /var/lib/postgresql/data directly
(bind-mount via docker-compose.zfs.yml override).

DBLab Engine (container: dblab_server, port 2345) reads the same ZFS dataset
and exposes thin-clone branches on ports 6000–6019 (localhost only).
```

v0.2/launch: move to a Hetzner Volume or Dedicated server with a separate disk
so the ZFS pool is not a loopback file on the OS disk.

### Provisioning — exact invocation order

Run these **as root on willbuy-v01**, in order:

```sh
cd /srv/willbuy

# 1. Create ZFS pool + datasets
sudo infra/zfs/setup-zfs-pgdata.sh

# Verify:
zfs list -r tank

# 2. Stop Postgres, migrate PGDATA to ZFS, restart
sudo infra/zfs/migrate-pgdata.sh

# Verify Postgres is back up:
docker compose -f infra/dev/docker-compose.yml \
  -f infra/dev/docker-compose.zfs.yml \
  --env-file infra/dev/.env -p dev \
  exec db pg_isready -U postgres

# 3. Install DBLab Engine (requires DBLAB_VERIFICATION_TOKEN in /etc/willbuy/secrets.env)
sudo DBLAB_VERIFICATION_TOKEN=$(grep DBLAB_VERIFICATION_TOKEN /etc/willbuy/secrets.env | cut -d= -f2) \
  infra/dblab/install-dblab.sh

# Verify DBLab health:
curl -s http://127.0.0.1:2345/healthz
```

Dry-run any step first with `--check`:

```sh
sudo infra/zfs/setup-zfs-pgdata.sh --check
```

### Starting the stack after provisioning

```sh
docker compose \
  -f /srv/willbuy/infra/dev/docker-compose.yml \
  -f /srv/willbuy/infra/dev/docker-compose.zfs.yml \
  --env-file /srv/willbuy/infra/dev/.env \
  -p dev \
  up -d
```

The `docker-compose.zfs.yml` override is generated by `migrate-pgdata.sh` and
replaces the `db-data` named volume with a bind-mount to `/var/lib/postgresql/data`.

### ZFS snapshot backup

DBLab takes snapshots automatically (every 6 h, retains 4 — see `infra/dblab/dblab.yml`).
For off-host backup, add a cron/systemd timer that runs:

```sh
# Send the latest snapshot to a remote host / object store
# (replace with Hetzner Storage Box or S3-compatible target)
SNAP=$(zfs list -t snapshot -H -o name tank/willbuy/pgdata | sort | tail -1)
zfs send "${SNAP}" | ssh backup-host "zfs receive backup/willbuy/pgdata"
```

This replaces any pg_dump cron (per spec §A5 constraint d).

### DBLab — creating and using thin clones

DBLab exposes an HTTP API on `127.0.0.1:2345`. Use the `dblab` CLI or the API directly.

```sh
# Create a clone (database branch)
dblab clone create --id pr-123 --username postgres --password 'SecurePass!'

# List all clones
dblab clone list

# Connect to the clone (port 6000 by default for the first clone)
psql -h 127.0.0.1 -p 6000 -U postgres -d postgres

# Destroy the clone when done
dblab clone destroy --id pr-123
```

### Using DBLab clones in CI / per-PR ephemeral databases

1. In your CI workflow (GitHub Actions), add a step that creates a DBLab clone:

```yaml
- name: Create DBLab clone
  run: |
    dblab init --url "$DBLAB_URL" --token "$DBLAB_TOKEN" --environment-id willbuy-ci
    CLONE_ID="pr-${{ github.event.number }}"
    dblab clone create --id "${CLONE_ID}" --username postgres --password "${DBLAB_CLONE_PW}"
    echo "DATABASE_URL=postgres://postgres:${DBLAB_CLONE_PW}@willbuy-v01:6000/postgres" >> "$GITHUB_ENV"
```

2. Run integration tests against `DATABASE_URL`.

3. After tests, destroy the clone:

```yaml
- name: Destroy DBLab clone
  if: always()
  run: dblab clone destroy --id "pr-${{ github.event.number }}"
```

Secrets needed in the CI environment:
- `DBLAB_URL` — `http://willbuy-v01:2345` (SSH tunnel or internal network)
- `DBLAB_TOKEN` — value of `DBLAB_VERIFICATION_TOKEN` from the 1Password vault
- `DBLAB_CLONE_PW` — a password for the clone's Postgres user

### VM upsize decision

CPX21 has **80 GB SSD** (README clarified from an earlier 40 GB draft — see Hetzner resource
table above). The loopback ZFS pool is capped at 30 GB, leaving ~50 GB for the OS + Docker layers.
This is acceptable for v0.1 (dev/staging, no production data volume yet).

**No VM upsize is required for PR #49.** Flag for v0.2 / launch:
- Move the ZFS pool to a Hetzner Volume (separate disk) for data integrity.
- Evaluate CPX31 (8 GB RAM) if DBLab clone count or Postgres shared_buffers grows.

## Secrets policy

- **Repo is public.** No secret ever enters this repo.
- Canonical source: **1Password vault `willbuy`** (personal account `my.1password.com`). Item titles match the `op://` paths in `.env.op`.
- Server-local cache: `/etc/willbuy/secrets.env`, mode 0600, owned by root. Written only by `scripts/push-secrets.sh` invoked through `op run --env-file=.env.op` — values are streamed over SSH stdin and never touch a shell variable, local file, or log.
- Containers read via `--env-file /etc/willbuy/secrets.env` or docker secrets.
- Preview envs get a dedicated sub-key per spec §2 #25.
- Any secret that appears in a chat/IM/email/commit message is treated as **burned** — rotate immediately.
